* Language Rules

** Emacs Lisp Rules

*** FILE ORDER
Header → Requires → Defgroup/Defcustom → Struct → Defvar → Functions → Provide

*** CODING RULES
| Rule     | Guideline                                      |
|----------+------------------------------------------------|
| DRY      | Extract helpers, no duplicate logic            |
| Comments | Docstrings yes, inline only for "why"          |
| Params   | 0-2 positional OK, 3+ use ~&key~               |
| Size     | <20 lines/function, single responsibility      |

*** LIBRARIES (prefer over reimplementing)
| Need          | Use                          |
|---------------+------------------------------|
| Lists         | ~dash.el~, ~map.el~          |
| Alists        | ~a.el~, ~asoc.el~            |
| Hash tables   | ~ht.el~                      |
| Strings       | ~s.el~, ~subr-x~             |
| Queues/Heaps  | ~queue.el~, ~heap.el~        |
| Async         | ~async.el~                   |
| Persistence   | ~persist.el~                 |

Perf: hash > alist/plist; plist >> alist (string keys); dash > cl-nset-difference

*** ANTI-PATTERNS
| Category       | Bad Practice                                                    |
|----------------+-----------------------------------------------------------------|
| Structure      | Mixed order, requires after code, missing lexical-binding       |
| Quality        | God functions (100+), debug msgs, too many positional params    |
| Naming         | Typos (~plsit-get~), accessor=class name, inconsistent prefixes |
| EIEIO          | Permissive types, wrong slot docs, mixed slot-value/accessors   |
| cl-loop        | ~on~ vs ~in~ confusion (on=cdrs, in=elements)                   |
| Error handling | No nil checks, silent ~(_ nil)~ in pcase, incomplete branches   |
| Docs           | "Interactively" without ~(interactive)~, wrong param names      |
| Smell          | ~(string-join (list x y))~ → use ~concat~, unused bindings, TODOs |
| State          | Multiple defvar-local duplicating object slots                  |
| Delegation     | Fallback code when delegating to objects                        |

*** SIMPLICITY RULES
| Rule | Guideline |
|------+-----------|
| One state source | One variable/object holds state, no duplicates |
| No fallbacks | Delegate directly, don't inline backup logic |
| No conditionals for dispatch | Use polymorphism or separate functions |
| Single path | One way to do something, not multiple |
